This is the single, final YAML — `import-smoke` uses `pytest tests/ -v` so all Wave-0 tests run in CI (test_import.py, test_hacs_manifest.py, and test_info_md.py — hacs-manifest and info-md are <0.5 s filesystem checks, so there is no speed penalty over the narrower `pytest tests/test_import.py -v`):

    ```yaml
    # Phase 1 CI skeleton — lint + import-smoke + hassfest manifest validation.
    # Phase 5 will add: HACS action (against GitHub mirror) + full pytest-HA-custom suite.

    stages:
      - lint
      - test
      - validate

    variables:
      PIP_CACHE_DIR: "$CI_PROJECT_DIR/.cache/pip"
      PYTHONDONTWRITEBYTECODE: "1"
      PIP_DISABLE_PIP_VERSION_CHECK: "1"

    default:
      image: python:3.12-slim
      cache:
        key: pip-$CI_COMMIT_REF_SLUG
        paths:
          - .cache/pip/

    ruff:
      stage: lint
      script:
        - pip install --quiet ruff==0.15.11
        - ruff check .
        - ruff format --check .

    import-smoke:
      stage: test
      script:
        - pip install --quiet pytest
        - pytest tests/ -v

    hassfest:
      stage: validate
      image: docker:25-cli
      services:
        - docker:25-dind
      variables:
        DOCKER_HOST: tcp://docker:2375
        DOCKER_TLS_CERTDIR: ""
      script:
        - docker run --rm -v "$CI_PROJECT_DIR:/github/workspace" ghcr.io/home-assistant/hassfest
    ```

    Rationale for exact content:
    - **stages order:** lint → test → validate. Fast fail on lint errors before paying for pytest or docker setup.
    - **ruff==0.15.11:** Latest pinned version as of 2026-04-20 (01-RESEARCH.md §Standard Stack). Pinning prevents a future ruff release from silently breaking CI.
    - **`ruff check .` AND `ruff format --check .`:** Both required — check catches lint issues, format --check catches formatting issues without modifying files.
    - **`pytest tests/ -v`:** Uses `pytest tests/ -v` (not just `test_import.py`) so all Wave-0 guard tests run in CI — hacs-manifest and info-md are <0.5 s filesystem checks; no speed penalty.
    - **hassfest job uses docker-in-docker (DinD):** python:3.12-slim doesn't have docker. DinD adds ~20s overhead per pipeline (01-RESEARCH.md §Code Examples notes). Use docker:25-cli client + docker:25-dind service. DOCKER_HOST + DOCKER_TLS_CERTDIR="" are the standard DinD environment vars. **The GitLab runner executing this pipeline MUST permit privileged mode** — see user_setup entry "gitlab-runner". If that is not possible for this project's runners, replace the hassfest job with the no-Docker fallback below.
    - **`ghcr.io/home-assistant/hassfest`:** Public image, no auth. linux/amd64 only (Pitfall 3); stock GitLab shared runners are x86_64 so this works out of the box.
    - **NO HACS action job in Phase 1:** Per 01-RESEARCH.md §Pitfall 2, the HACS action cannot validate a GitLab-only repo. Phase 5 adds it against the GitHub mirror.

    **Fallback: no-DinD hassfest (only if the runner cannot be put into privileged mode):**

    If the user confirms in the user_setup checkpoint that the runner cannot permit DinD, swap the `hassfest` job for this scripted variant. It runs hassfest as a Python script under the same python:3.12-slim image — slower (first run installs homeassistant, ~45–60 s), but requires no Docker:

    ```yaml
    hassfest:
      stage: validate
      script:
        - pip install --quiet homeassistant
        - python -m script.hassfest --action validate --integration-path custom_components/party_dispenser
    ```

    Do NOT apply this swap pre-emptively — default to the DinD variant (faster, canonical). Apply the fallback only after the user confirms their runner does not permit privileged mode.
